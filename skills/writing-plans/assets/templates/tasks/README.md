# Tasks Guide

这个目录用于把研发目标落成可执行任务。目标不是记流水账，而是持续写清楚：
- 要做什么。
- 为什么现在做。
- 做到什么算完成。
- 依赖什么。
- 完成后解锁什么。
- 当前卡在哪里。

## 目录结构

```text
tasks/
├── README.md                # 本指南
├── index.md                 # 总任务视图 / dashboard
├── inbox.md                 # 临时记录 / capture
├── tasks/                   # 单个任务卡
├── projects/                # 项目级说明与里程碑
└── templates/               # 任务模板
```

推荐使用顺序：
- 先看 `index.md` 了解优先级和关键路径。
- 再看 `projects/<project>.md` 了解项目约束、里程碑和最终验收目标。
- 开始具体工作前，进入对应 `tasks/task-xxx.md`。
- 临时想法、待整理问题先放 `inbox.md`，不要把未成型内容塞进正式任务卡。

## 任务系统原则

- `Frontmatter` 负责机器可读：状态、优先级、依赖、项目归属。
- `Markdown Body` 负责人类可读：背景、目标、步骤、验收、日志、备注。
- 任务卡必须可执行，不写空泛口号。
- 每张任务卡必须能回答 4 个问题：
  - 为什么现在做。
  - 做到什么算完成。
  - 依赖什么任务。
  - 完成后解锁什么任务。
- 任务服务于最终目标，不服务于固定流程；一旦发现当前任务定义、依赖链或实现路径不再合理，必须及时回写并调整。

## 当前项目约束

这里写当前项目的强约束。典型内容包括：
- 是否需要兼容旧实现。
- 是否允许破坏性重构。
- 文件规模限制。
- 模块化/拆包边界。
- 构建产物同步要求。
- 团队明确约定的技术选择优先级。

如果没有约束，不要写空话；保持这一节简洁。

## 状态约定

- `todo`
  - 任务已定义，但尚未开始。
- `doing`
  - 当前正在推进，并且应该在 `index.md` 的 `Doing` 区出现。
- `done`
  - 已完成，并满足该卡的“任务完成验收标准”。
- `blocked`
  - 因外部依赖或前置任务未完成而无法继续。

状态切换规则：
- `todo -> doing`
  - 开始实际执行时再切换，不是“打算做”。
- `doing -> done`
  - 必须先满足验收标准，再改状态。
- `doing -> blocked`
  - 必须在日志里写清楚阻塞点和恢复条件。
- `blocked -> doing`
  - 阻塞解除后恢复推进，并补日志。

## 优先级约定

- `P0`
  - 关键路径任务；不完成会阻塞主线。
- `P1`
  - 重要但不一定卡主线；通常是补齐闭环、恢复能力或工具链。
- `P2`
  - 后续阶段或非首批闭环内容。

如果优先级和依赖冲突，以依赖链为准，不要靠提高优先级跳过前置能力。

## Frontmatter 字段说明

```yaml
id: task-001
title: 替换成可执行任务标题
status: doing
priority: P0
tags: [architecture]
project: your-project-slug
due: 2026-04-22
parent: null
depends_on: [task-000]
blocks: [task-002]
```

字段解释：
- `id`
  - 全局唯一，格式固定为 `task-xxx`。
- `title`
  - 必须是可执行任务，不写“继续优化”“完善一下”。
- `status`
  - 只能使用 `todo | doing | done | blocked`，或你的系统里明确约定的状态集合。
- `priority`
  - 默认使用 `P0 | P1 | P2`。
- `tags`
  - 用于快速按领域归类，例如 `runtime`、`api`、`docs`、`tooling`。
- `project`
  - 所属项目 slug。
- `due`
  - 计划目标日期，不是承诺日期；过期后要调整，不要静置。
- `parent`
  - 若存在上级任务则填任务 ID，否则 `null`。
- `depends_on`
  - 当前任务开始或完成前必须依赖的任务。
- `blocks`
  - 当前任务完成后会直接解锁的任务。

## 任务卡正文规范

每张任务卡必须包含以下部分：
- `背景`
- `目标`
- `执行步骤`
- `验收标准`
- `执行中发现与动态调整`
- `执行日志`
- `Review`
- `备注`

约束如下：
- `背景`
  - 说明现状问题和这个任务存在的必要性。
- `目标`
  - 只写任务完成后应达到的结果，不写执行过程。
- `执行步骤`
  - 过程型条目，能被勾选。
- `验收标准`
  - 分成两层：
  - `步骤验收标准`
  - `任务完成验收标准`
- `执行中发现与动态调整`
  - 记录执行中发现的问题、更优方案、失效假设、潜在损失和对后续任务的影响。
- `执行日志`
  - 记录时间、进展、阻塞、决策、结果。
- `Review`
  - 任务完成前必须复盘是否真的推进最终目标，以及是否需要调整后续任务、依赖、优先级和验收标准。
- `备注`
  - 放风险、假设、关联文件、坑点。

## 如何写执行步骤

好的步骤应该：
- 按依赖顺序排列。
- 每一步最好对应一个可观察输出。
- 能被验收，不要写“研究一下”“看看能不能”。

不好的例子：
- [ ] 优化整体架构
- [ ] 处理一些边界情况

好的例子：
- [ ] 定义 API 数据结构与校验规则
- [ ] 为错误输入补一个最小复现测试并让它通过
- [ ] 把旧入口迁移到新 SDK 并删除 raw 调用

## 如何写验收标准

步骤验收标准：
- 对应每一个执行步骤。
- 必须能观察、验证、复现。
- 优先写结果，少写主观描述。

任务完成验收标准：
- 回答“这张任务卡到底什么时候能改成 done”。
- 面向后续任务，而不是只面向当前提交。
- 明确排除“代码写了一半、文档没更新、测试没补齐也算 done”这种情况。

推荐写法：
- 步骤 1 验收：新增接口可以被完整解析，并区分核心字段与展示字段。
- 任务完成验收标准：调用方已能基于新接口完成预期业务闭环，不再依赖旧实现。

## 执行日志规则

- 每次开始做实质工作、完成关键节点、遇到阻塞，都应更新日志。
- 日志优先写事实：
  - 做了什么。
  - 发现什么问题。
  - 做了什么决策。
  - 下一步是什么。
- 不写无信息量内容，例如“继续开发”“处理了一些问题”。

推荐格式：

```markdown
## 2026-04-22
- 完成接口定义与校验规则补齐。
- 发现旧调用链仍直接依赖旧字段，需要在下一步切断。
```

## 执行中发现与动态调整规则

执行中发现以下任一情况，不能只记在聊天或脑中，必须写进当前任务卡的 `执行中发现与动态调整`：
- 当前实现路径存在明显问题、隐藏风险或可能造成不可逆损失。
- 出现比原计划更稳妥、更低成本或更符合目标架构的方案。
- 上游假设失效，导致当前任务或后续任务定义不再准确。
- 继续机械照着原步骤推进，会导致返工、错误沉淀或方向越走越深。

记录时至少写清楚：
- 发现了什么。
- 为什么它是问题，或为什么新方案更好。
- 会影响哪些后续任务、依赖、验收或里程碑。
- 建议动作：继续、拆卡、改依赖、改优先级、改验收、暂停。

如果影响关键路径、最终验收或多个后续任务，不能只改当前任务卡，要同步更新 `index.md` 和项目页。

## Review 规则

- 每张任务在改成 `done` 前必须完成 `Review`。
- Review 不是重复总结，而是确认：
  - 当前结果是否真的服务最终验收目标。
  - 当前实现是否暴露了新风险、新依赖或更合理的拆分方式。
  - 后续任务的标题、目标、依赖、优先级、验收标准是否仍然成立。
- Review 结论至少要包含：
  - 当前任务是否允许标记为 `done`。
  - 是否需要调整后续任务；如果不需要，也要明确写“无需调整”。
  - 如果需要调整，具体改了什么、为什么改。

如果 Review 判断后续任务需要变化，应直接修改对应任务卡、`index.md`、必要时更新项目页；任务只是手段，不是准则。

## Dashboard 维护规则

`index.md` 不是自动生成的，采用半自动维护。

更新规则：
- 某任务进入 `doing`，同步更新 `index.md` 的 `Doing`。
- 某任务完成，移动到 `Done`。
- 关键路径变化、里程碑变化、最终验收变化时，要同步更新 `index.md`。
- 如果出现显式阻塞，写入 `Blocked`，不要只停留在单个任务卡中。
- 如果任务的 `Review` 调整了后续任务、依赖链或优先级，也要同步更新 `index.md`。

## Inbox 使用规则

`inbox.md` 只放临时内容：
- 突然发现的风险
- 未分配归属的问题
- 暂时没拆成任务的想法
- 执行中冒出来但还不确定是否成立的 follow-up

定期整理原则：
- 能归类到现有任务卡，就迁回对应任务。
- 能独立成任务，就新建任务卡。
- 无效或过时内容直接删除。

## 新建任务的时机

需要新建任务卡的情况：
- 现有任务明显过大，无法在一个稳定目标下推进。
- 新工作拥有独立目标、独立验收标准、独立依赖。
- 某风险已经从“备注”升级为需要正式跟踪的工作项。

不需要新建任务卡的情况：
- 只是当前任务中的一个子步骤。
- 只是一次局部修正，不会形成新的依赖链。
- 只是日志级别的信息更新。

## 依赖关系写法

- `depends_on`
  - 写“没有它就没法开始或没法完成”的硬依赖。
- `blocks`
  - 写“我完成后会直接解锁它”的下游任务。

不要滥写依赖：
- 如果只是有关联，不一定阻塞，不要写进 `depends_on`。
- 如果只是可能受影响，但不必等待，不要写进 `blocks`。

## 何时可以标记为 done

同时满足以下条件才能标记 `done`：
- 执行步骤已完成，或未完成的步骤已明确证明不再需要。
- `步骤验收标准` 已满足。
- `任务完成验收标准` 已满足。
- `执行中发现与动态调整` 已记录清楚所有关键偏差、问题和新方案。
- `Review` 已完成，且后续任务所需调整已经同步。
- 相关文档、日志、dashboard 已同步。
- 没有遗留会立即阻断下游任务的关键缺口。

## 与最终验收的关系

这个任务系统最终不是为了“把基础设施做完”或“把任务都关掉”，而是为了完成项目级最终验收。你需要在项目页和 `index.md` 中写清楚最终验收信号，并用它们约束优先级。

如果某项工作看起来很完整，但对最终验收没有贡献，就应该重新审视它的优先级。

## 推荐工作流

1. 从 `index.md` 选择当前优先任务。
2. 打开对应任务卡，确认依赖、目标和验收标准。
3. 开始实施前，补充或修正执行步骤与验收标准。
4. 实施过程中持续更新 `执行日志`，并把发现的问题、风险和更优方案写进 `执行中发现与动态调整`。
5. 完成后对照验收标准逐项确认。
6. 填写 `Review`，判断后续任务、依赖、优先级和验收是否需要调整。
7. 同步更新 `index.md`、必要时更新项目页和受影响任务卡。
8. 如果出现新增工作，先写到 `inbox.md`，再决定是否拆卡。
